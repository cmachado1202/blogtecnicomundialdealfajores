ENTREGA FINAL · BLOG TÉCNICO

Cómo mejoré el carrito de compras de un e-commerce sin perder el foco en la experiencia de compra

Un caso práctico sobre catálogo dinámico, persistencia de datos, validación de stock y aprendizaje a partir de un incidente técnico.

Proyecto: Mundial de Alfajores • Lectura: 4 min • Actualizado: julio de 2026
TL;DR

Detecté que el carrito podía perder o recuperar mal información al recargar la página. Se ajustó la lectura de localStorage, se validó el stock antes de sumar productos y se documentó el incidente para prevenir que vuelva a ocurrir.

01 / CONTEXTO

Un e-commerce con catálogo dinámico y carrito persistente

El proyecto consiste en una tienda online ficticia para el Mundial de Alfajores. La página muestra productos obtenidos desde una API, permite agregarlos a un carrito, ajustar cantidades, eliminar ítems y conservar la selección aunque la persona usuaria actualice la página.

CatálogoDatos obtenidos desde API
CarritoEstado persistido en localStorage
ObjetivoCompra clara y sin cantidades inválidas

02 / PROBLEMA

El carrito no siempre recuperaba correctamente los productos

Durante las pruebas detecté que, después de actualizar la página, el carrito podía no recuperar los productos guardados o mostrar cantidades que superaban el stock disponible. Esto afectaba la continuidad de la compra y hacía que el comportamiento no fuera confiable para la persona usuaria.

⚠
Impacto

La sesión de compra podía quedar incompleta y el total podía no representar una cantidad válida de productos.

03 / INVESTIGACIÓN Y ACCIONES

Qué revisé y cómo lo resolví

Primero separé el problema en hipótesis simples y prioricé las que tenían mayor impacto sobre la compra. La revisión se centró en la carga inicial del carrito, la conversión de datos guardados y la lógica que incrementa cantidades.

  1. 01
    Revisé los datos guardados.

    Verifiqué qué ocurría cuando localStorage no tenía datos, contenía una versión anterior del carrito o devolvía un valor con formato inválido.

  2. 02
    Agregué una carga segura.

    Incorporé una validación con valor por defecto y manejo de errores para que el sitio pueda iniciar incluso si no hay datos almacenados.

  3. 03
    Validé el stock antes de modificar el carrito.

    La cantidad solo aumenta si todavía existe stock disponible. Cuando no lo hay, la interfaz muestra un mensaje claro.

  4. 04
    Probé el flujo completo.

    Realicé pruebas agregando, eliminando, modificando cantidades y recargando la página en pantalla de escritorio y móvil.

04 / POST-MORTEM CONSTRUCTIVO

Aprender del incidente, sin buscar culpables

En lugar de atribuir el problema a una persona, el análisis se enfocó en el proceso. La causa principal fue no contemplar todos los estados posibles de los datos guardados en el navegador y no validar el límite de stock desde el inicio del flujo.

Qué funcionó

Dividir la investigación en pasos pequeños permitió llegar rápido a una hipótesis verificable.

Qué faltaba

Casos de prueba para datos vacíos, dañados y cantidades mayores al stock.

Qué cambia

El checklist de pruebas incluye persistencia, stock y recarga antes de cerrar un ticket.

05 / CONTROL DE VERSIONES

Cambios trazables y fáciles de revisar

Los cambios se organizaron por objetivo para facilitar la revisión. En un repositorio real, cada ajuste queda asociado a un commit descriptivo y, cuando corresponde, a un Pull Request con contexto, pruebas realizadas e impacto esperado.

COMMIT feat: persistir y recuperar el carrito de forma segura ↗ COMMIT fix: validar stock al agregar productos al carrito ↗ REPO Repositorio público del proyecto ↗

06 / APRENDIZAJES

Mejoras que quedan para el próximo cambio

  • ✓ Validar datos guardados antes de usarlos en la interfaz.
  • ✓ Definir límites de stock desde el flujo de alta al carrito.
  • ✓ Documentar escenarios de prueba y resultados en cada ticket.
  • ✓ Usar commits específicos para poder revisar y revertir cambios con seguridad.

07 / FEEDBACK RADICALMENTE SINCERO

La mejora técnica también necesita empatía

Durante la revisión, el feedback se centró en el comportamiento del código y en el impacto sobre la compra, no en la persona que lo desarrolló. En lugar de señalar “esto está mal”, el comentario propone qué escenario falta cubrir, por qué importa y una alternativa concreta. Esta forma de revisar permite ser directa sobre la calidad sin perder respeto ni motivación.

“La lógica funciona bien en el flujo principal. Para evitar una compra inválida, ¿podemos validar el stock antes de incrementar la cantidad y sumar una prueba para ese caso?”

Una documentación breve, clara y honesta transforma un error puntual en una mejora reutilizable para todo el equipo.
